🔄 Python Concurrency

Asynchronous Python: Forms of Concurrency

Parallelism · Threading · Asyncio

Why it matters: asyncio.to_thread for blocking work from async, ThreadPoolExecutor for parallel sync tasks. Concurrency patterns in real applications.

Roadmap: Core terms → GIL limits → choosing the model → code examples → hands-on exercises and quiz.

Definitions & basic terms

Sync vs Async

Sync: tasks run one after another, blocking.

Async: tasks can start/end independently, non-blocking.

Concurrency vs Parallelism

Concurrency: multiple tasks make progress together (may alternate on one core).

Parallelism: tasks run simultaneously on multiple cores.

Terms: Blocking — caller waits. Thread — shared memory. Process — separate memory. Coroutine — async/await. Event loop — runs coroutines.

Parallelism implies concurrency. Concurrency does not always imply parallelism.

With terms clear, the next question is: why don't Python threads speed up CPU-heavy tasks?

GIL: Python's «Guard»

The Python interpreter is a kitchen with one chef (thread). GIL is the strict head chef who won't let others in while one is working.

  • CPU-bound (math, ML): GIL turns multithreading into an illusion
  • I/O-bound (network, API): GIL releases the thread during waiting

Brief facts about GIL:

  • Only one thread executes at a time; interpreter switches between threads for concurrency
  • GIL applies to CPython; Jython and IronPython have no GIL
  • GIL makes single-threaded programs fast; I/O usually not hindered
  • Enables non-thread-safe C libraries → many high-performance C extensions
  • CPU-bound: interpreter checks every N ticks, switches threads so one doesn't block others

Myth: «GIL only in Python». Ruby (MRI) has a similar mechanism.

Before coding with threads, we must understand the most common bug: race conditions.

Race condition

When two or more threads access shared data and at least one modifies it — outcome depends on timing, not just logic.

counter = 0
def increment():
    global counter
    counter += 1  # read → modify → write (not atomic!)

# 2 threads × 1000 calls each → expect 2000, often get less
  • Fix: Lock, atomic ops, or avoid shared state (Queue)

Now let's look at the synchronization tools that prevent this in real systems.

Thread sync & Queue

  • Lock — one thread at a time; prevents race conditions
  • Semaphore — limit N threads accessing a resource
  • Event — one thread waits for a signal from another
  • Queue — thread-safe data passing; prefer over shared variables (Effective Python)

Pipeline: download → parse → save via queues. Minimize shared state.

Cheat Sheet

Don't chase trends. asyncio isn't a «silver bullet». 10 requests → threading is simpler. multiprocessing — for heavy compute. Measure before optimizing!

  • LLM/API chat → asyncio
  • Parallel sync LLM → ThreadPoolExecutor
  • Sync DB from async → asyncio.to_thread
  • CPU-bound in async → asyncio.to_thread
  • Profile first: cProfile, time — measure before optimizing

2️⃣ Code Challenge — Predict the Output

What will be printed?

import threading
counter = 0
def increment():
    global counter
    for _ in range(100000):
        counter += 1
threads = []
for _ in range(2):
    t = threading.Thread(target=increment)
    threads.append(t)
    t.start()
for t in threads:
    t.join()
print(counter)

Why: counter += 1 is not atomic → race condition.

Fix version 
lock = threading.Lock()
def increment():
    global counter
    for _ in range(100000):
        with lock:
            counter += 1

4️⃣ Live Demo — I/O Concurrency

No concurrency

def task():
    time.sleep(2)
for _ in range(5):
    task()
# Result ≈ 10 seconds

With threads

threads = []
for _ in range(5):
    t = threading.Thread(target=task)
    threads.append(t)
    t.start()
for t in threads:
    t.join()
# Result ≈ 2 seconds

💡 Perfect demonstration of I/O concurrency.

5️⃣ Trick Question

Why doesn't this run faster with threads?

def cpu_task():

for _ in range(4):
    t = threading.Thread(target=cpu_task)

📚 Effective Python — Threads & Concurrency

  • 1. Prefer Queue — queue.Queue for thread-safe data passing; avoids race conditions; producer/consumer pattern.
  • 2. Threads for I/O-bound — network, disk, API, DB. During I/O wait the GIL is released.
  • 3. Avoid threads for CPU-bound — use multiprocessing or ProcessPoolExecutor.
  • 4. Use concurrent.futures — ThreadPoolExecutor / ProcessPoolExecutor instead of low-level threading; less boilerplate.
  • 5. Minimize shared state — prefer message passing (queues); use locks only when necessary. Beware race conditions and deadlocks.
  • 6. Pipelines — download → parse → save via queues; improves I/O throughput.

Summary: I/O → threads · CPU-heavy → multiprocessing · Communication → queues · Abstraction → ThreadPoolExecutor
